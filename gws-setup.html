Setting up gws on a new person's machine

Give Dad (or anyone) the Google Workspace CLI wired to his own Google account and his own Google Cloud project — a mirror of the working setup, built from scratch on his machine. Nothing is copied from Reut's computer.

🔑 The one rule

He gets his own credentials — his own GCP project, his own OAuth client, authenticated as his Google account. Never copy Reut's client_secret.json or tokens onto his machine. Everything below runs on his computer, signed into his account.

0 · Prerequisites

Run this on his machine first. It tells you which path Step 1 takes.

brew --version && gws --version && gcloud --version
  • No gws → install it:
brew install googleworkspace-cli
  • No gcloud → install it for the one-command path (or skip it and use the no-gcloud path):
brew install --cask google-cloud-sdk

1 · Create his project + Desktop OAuth client

gws auth setup creates the Cloud project, enables the APIs, and makes the Desktop OAuth client in one shot — it's what produced gws-cli-reutaux originally.

# sign in as HIS account — see trap #1 below
gcloud auth login

# creates gws-cli-<his>, enables APIs, makes the Desktop OAuth client
gws auth setup

2 · Publish the consent screen to Production

This is console-only — it can't be done from the CLI — and it's the fix for the token dying every 7 days. Do it as his account. Replace the placeholder with his real project id from Step 1.

https://console.cloud.google.com/auth/audience?project=gws-cli-his-localpart

Publishing status Testing → Publish app → Confirm. Published-unverified is fine for personal use; the 100-user cap is irrelevant.

3 · Log him in

The OAuth browser flow — he approves scopes as himself.

gws auth login

TRAP Never copy-paste the printed URL

Terminal line-wrap truncates the long OAuth URL → Error 400: invalid_request — missing response_type. Cmd-click it instead. If Claude is driving on his machine, background the login and hand over a clickable link:

pkill -f "gws auth login"            # clear stale login servers
gws auth login > /tmp/gws-login.log 2>&1 &   # background it, then read the full URL from the log

Append &login_hint=his-email to the URL to pre-pick his account. At Google's "hasn't verified this app" screen → Advanced → Go to gws-cli-his (unsafe). That's expected, because it's unverified-Production.

4 · Verify

gws auth status

token_valid: true, and project_id should be his, not gws-cli-reutaux. Then a real call:

gws calendar events list --params '{"calendarId":"primary","maxResults":1}'

The 3 things that weren't straightforward

All three bit the original setup. They'll all show up at his too.

1 Wrong Google account auto-validates

If he's signed into more than one Google account in the browser, the chooser silently grabs the wrong one. Explicitly click his account at every step — the gcloud auth login, and the OAuth approval. This caused more grief than anything else last time.

2 Testing-mode = weekly re-auth

Skip Step 2 and his refresh token dies every ~7 days with invalid_grant: Token has been expired or revoked. Publishing the consent screen to Production once is the whole fix.

3 Truncated login URL

Copy-pasting the login URL out of the terminal breaks it (missing response_type). Cmd-click it, or background-and-clickable-link as in Step 3.

No-gcloud path — create the project by hand in the Cloud Console

Same end result as Step 1, all clicks, no gcloud install. Do it all as his account at console.cloud.google.com.

  1. New project — name it something like gws-cli-his-localpart.
  2. Enable these 9 APIs (APIs & Services → Library):
drive.googleapis.com
gmail.googleapis.com
calendar-json.googleapis.com
docs.googleapis.com
sheets.googleapis.com
slides.googleapis.com
tasks.googleapis.com
people.googleapis.com
meet.googleapis.com
  1. Configure the OAuth consent screen → User type External → fill app name + his email.
  2. Credentials → Create Credentials → OAuth client ID → Application type: Desktop app.
  3. Download JSON and save it exactly here:
~/.config/gws/client_secret.json
  1. Then do Step 2 (publish to Production) and Step 3 (gws auth login).
Every command block copies on click. Placeholders like his-localpart and his-email are the only things you swap in.
